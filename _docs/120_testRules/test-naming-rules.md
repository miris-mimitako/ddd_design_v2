# Test Naming Rules

## 基本

テスト名は、何をしたらどうなるかが読める形にする。

推奨形式:

- `<条件>なら<結果になる>`
- `<前提>のとき<振る舞う>`
- `<ユースケース>すると<結果を返す>`

例:

- 18歳以上なら登録条件を満たす
- 未承認注文のとき出荷を拒否する
- 新規登録すると保存してイベントを発行する

## 避ける名前

- test1
- 正常系
- 異常系
- saveが呼ばれる
- works correctly

これらは意図ではなく実装や分類しか示さない。

## describe / context の使い方

- describe は業務概念またはユースケース
- context は前提条件
- it は期待される振る舞い

例:

- describe: UserRegistration
- context: 既存メールアドレスが存在しないとき
- it: ユーザーを保存して成功を返す

## ID の置き方

ID は次のどちらかで持つ。

- テストケース台帳の `id`
- テストコードのコメントまたは metadata

例:

`BR-002`, `T-APPLICATION-USER-001`

ID をテスト名そのものに無理に埋め込む必要はない。
