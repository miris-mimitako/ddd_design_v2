# Core Business Purpose

## このディレクトリの役割

ここには、業務ルールそのものではなく、**この業務や機能がなぜ存在するか**を書く。

言い換えると、ここで扱うのは次のような内容である。

- 何を達成したいのか
- 誰にとって価値があるのか
- 何を失敗とみなすのか
- どの指標で成果を判断するのか
- どこまでを対象範囲とするのか

業務ルールの詳細な制約や例外は `10_businessRules` に置く。
用語定義は `20_ubiquitousLanguage` に置く。
振る舞いの具体例は `120_testRules` に従ってテストへ寄せる。

## 書くべき内容

### 1. 業務目的

この機能や業務が何を改善するのかを書く。

例:

- 登録業務の手戻りを減らす
- 誤出荷を防ぐ
- 通知漏れを防ぐ
- 審査の一貫性を高める

### 2. 対象ユーザーと価値

誰がこの機能を使い、どんな価値を得るかを書く。

例:

- 利用者は短時間で正しい登録を完了できる
- 管理者は監査可能な状態で業務を処理できる

### 3. 成功条件

何をもって成功とみなすかを書く。

例:

- 条件を満たす入力は迷いなく完了できる
- 重要な業務違反は事前に防止される
- 監査時に判断根拠を追跡できる

### 4. 失敗コスト

この機能が誤ると何が起きるかを書く。

例:

- 不正登録が発生する
- 誤出荷が発生する
- 通知先の整合性が崩れる
- 監査で説明できない

### 5. 対象範囲

どこまでをこの業務目的の範囲に含めるかを書く。

例:

- 新規登録の成立条件までは対象
- DB 実装方法は対象外
- 画面レイアウト詳細は対象外

## 書かないもの

ここには次を書かない。

- 具体的な業務ルール一覧
- 正常系、異常系、境界値の振る舞い
- 画面遷移の詳細
- API 形式
- DB 設計
- 実装手順

## 記述テンプレート

```md
# <業務名または機能名>

## Purpose
- 何を達成したいか

## Target User
- 誰に価値があるか

## Success Criteria
- 何をもって成功とみなすか

## Failure Cost
- 何を失敗とみなすか

## Scope
- 何を対象に含めるか
- 何を対象外にするか

## Related Documents
- Business Rules: ...
- Ubiquitous Language: ...
- Test Rules: ...
```

## サンプル

### ユーザー登録

#### Purpose

- 条件を満たす利用者だけを一貫した基準で登録できるようにする。
- 登録後の認証、通知、監査で矛盾が出ない状態を作る。

#### Target User

- 利用者
- 登録内容を管理する運用担当者

#### Success Criteria

- 条件を満たす利用者は迷いなく登録できる。
- 条件を満たさない利用者は業務上妥当な理由で拒否される。
- 登録判断の前提となる用語と条件が追跡できる。

#### Failure Cost

- 不適切な利用者が登録される。
- 通知先や本人性の整合性が崩れる。
- 後続業務で修正コストが増える。

#### Scope

- 新規登録の業務目的と成功条件を対象に含める。
- 年齢条件や重複禁止の具体ルールは対象外とし、`10_businessRules` で扱う。
- 判定の具体的振る舞いは対象外とし、テストで扱う。
