# Document Writing Rules

## 目的

この文書は、ドキュメントの書き方を統一するための基準である。

重要なのは、文書を増やすことではなく、**読んだ人が意図を誤解しないこと**である。

## 基本原則

- 文書は意図と制約を書く
- 振る舞いの詳細はテストに寄せる
- 実装方法は書きすぎない
- 1文書1責務にする
- 用語はユビキタス言語に合わせる

## 文体ルール

- 断定で書く
- 主語を省略しすぎない
- 曖昧語を避ける
- 技術者だけに通じる略語を多用しない
- 1項目に複数のルールを詰め込まない

避ける表現:

- なるべく
- 必要に応じて
- 基本的には
- 適切に
- いい感じに

これらの語を使う場合は、条件を続けて明記する。

## 記述単位

1つの項目には次だけを書く。

- ルール本体
- 背景
- 例外
- 関連ID

1つの項目に書かないもの:

- 実装フロー全文
- テストケース全文
- UI 仕様の全詳細

## ルール記述テンプレート

業務ルールは、次の粒度で書く。

```text
ID:
Rule:
Reason:
Exception:
Related Terms:
Related Tests:
```

例:

```text
ID: BR-001
Rule: 注文は承認済みでなければ出荷できない。
Reason: 誤出荷防止と監査要件のため。
Exception: 管理者による監査解除手続きが完了している場合を除く。
Related Terms: 注文, 承認済み, 出荷
Related Tests: T-DOMAIN-SHIP-001
```

## 例の扱い

例は理解補助として使う。
例が仕様本体になってはいけない。

良い例:

- ルールの意味を補強する最小限の例

悪い例:

- あらゆる分岐を例で列挙する
- テストで表現すべきケースを文書に書き切る

## 実装詳細の扱い

次は文書の主内容にしない。

- テーブル構造
- HTTP パラメータ名
- ORM マッピング方法
- クラス分割の都合
- フレームワークの制約回避策

必要なら `110_codingRules` かコードコメントに寄せる。

## 図の扱い

図は関係性を説明するために使う。
図で振る舞いの全ケースを表現しようとしない。

図が向いているもの:

- 境界づけられたコンテキスト
- 集約の責務境界
- 依存方向
- システム間接続

図が向かないもの:

- 分岐条件の完全仕様
- 期待値の列挙

## 更新ルール

文書更新時は次を確認する。

1. 用語が既存定義と衝突していないか
2. 振る舞いを文書に書きすぎていないか
3. テスト更新が必要なのに漏れていないか
4. 技術都合が業務ルールへ混入していないか

## 完了条件

文書が完成したとみなす条件は次の通り。

- 背景と制約が読める
- 用語が一貫している
- テストへ落とすべき内容が分離されている
- 実装詳細が過剰に書かれていない
