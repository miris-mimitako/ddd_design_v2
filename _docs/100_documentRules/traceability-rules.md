# Traceability Rules

## 目的

この文書は、ドキュメント、テスト、コードを弱く接続するためのルールを定義する。

目的は追跡可能性の確保であり、管理表を増やすことではない。

## 基本方針

- 文書とテストを参照可能にする
- ただし 1対1 対応を強制しない
- 粒度の違いを許容する
- 人が追えることを優先する

## ID 体系

### 業務目的

- `BP-xxx`

例:

- `BP-001`

### 業務ルール

- `BR-xxx`

例:

- `BR-014`

### 用語

- `TERM-xxx`

例:

- `TERM-007`

### 非機能要件

- `NFR-xxx`

例:

- `NFR-003`

### ユースケース

- `UC-xxx`

例:

- `UC-010`

### テスト群

- `T-<layer>-<topic>-xxx`

例:

- `T-DOMAIN-ORDER-001`
- `T-APPLICATION-SHIPMENT-002`
- `T-CONTRACT-USER-001`

## 関連付けルール

### 文書からテストへの参照

業務ルールやユースケースには、関連テスト ID を持たせてよい。

例:

```text
ID: BR-001
Related Tests: T-DOMAIN-SHIP-001, T-APPLICATION-SHIP-003
```

### テストから文書への参照

テスト名、describe、コメント、metadata のいずれかで関連 ID を示してよい。

例:

```text
BR-001: 注文は承認済みでなければ出荷できない
```

## 粒度ルール

- 1つの業務ルールに複数テストが紐づいてよい
- 1つのテストが複数ルールを検証してもよい
- ただし、どのルールを主に確認しているかは分かるようにする

## 参照すべき対象

トレース対象に含めるもの:

- 業務目的
- 業務ルール
- 用語
- 非機能要件
- ユースケース
- テスト群

トレース対象に含めないもの:

- private メソッド単位の実装
- 細かすぎる補助関数
- フレームワーク設定値

## 運用ルール

仕様変更時は次の順で確認する。

1. 変更対象の `TERM` を確認する
2. 影響する `BR` と `UC` を確認する
3. 関連する `T-*` を更新する
4. 必要なら `NFR` への影響を見る

## 禁止事項

- 追跡表のためだけに意味のない ID を大量発行しない
- 実際に使われない参照リンクを機械的に増やさない
- テストを持たない重要ルールを放置しない
- 文書上で関連すると書いたのにテストが存在しない状態を常態化させない

## 良い状態

良い状態は次の通り。

- 重要な業務ルールから関連テストへ辿れる
- 失敗したテストから関連ルールへ辿れる
- 用語変更時に影響範囲を追える
- 実装詳細に潜らなくても仕様の芯へ到達できる
