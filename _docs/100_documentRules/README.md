# Document Rules

## このディレクトリの役割

`100_documentRules` には、業務仕様そのものではなく、**仕様をどこに残すかのルール**を置く。

このプロジェクトでは、情報を次の3つに分離する。

- 意図と制約はドキュメントに残す
- 振る舞いはテストに落とす
- 実現方法はコードに閉じ込める

このディレクトリは、その分離を崩さないための基準を定義する場所とする。

## ここに入れるべき内容

### 1. 文書配置ルール

どの情報をどのディレクトリに置くかを定義する。

- `00_coreBusinessPurpose`
  - 事業目的
  - 成果指標
  - なぜこの業務が必要か
- `10_businessRules`
  - 業務ルール
  - 禁止事項
  - 例外条件
  - 受け入れ条件
- `20_ubiquitousLanguage`
  - 用語定義
  - 同義語禁止
  - UI名、コード名、DB名の対応
- `100_documentRules`
  - 文書に何を書くか
  - 文書に何を書かないか
  - 文書とテストの責務分離
- `110_codingRules`
  - 実装方針
  - レイヤ依存ルール
  - 命名規則
- `120_testRules`
  - テスト戦略
  - テスト粒度
  - テスト命名

### 2. 文書に残す情報の基準

文書には、次のような**意味の説明が必要なもの**だけを残す。

- 業務上の目的
- 業務ルールの背景
- 用語の定義
- 禁止事項
- 例外条件
- 非機能要件
- 監査要件
- 認可要件
- テストでは読み取りにくい判断理由

逆に、次のような**実行可能なもの**は文書よりテストを正本に寄せる。

- 正常系の振る舞い
- 異常系の振る舞い
- 前提条件と結果の対応
- ユースケースの処理結果
- ポート契約の入出力

### 3. 文書に書いてはいけないもの

重複を防ぐため、以下は原則としてここに書かない。

- 実装手順
- 画面遷移の細かい分岐仕様
- SQL や ORM 前提の説明
- API レスポンスの細部
- テストコードにそのまま落とせる具体例の羅列
- コードを読めば分かる private な実装都合

判断基準は単純で、**その内容がテストで読めるなら文書で重複しない**。

### 4. 文書とテストの責務分離

文書は「なぜそうするか」を説明する。
テストは「どう振る舞うか」を固定する。

例:

- 文書
  - 注文は承認済みでなければ出荷できない
  - 理由は監査と誤出荷防止
- テスト
  - 承認済み注文は出荷できる
  - 未承認注文は出荷できない
  - 取消済み注文は出荷できない

### 5. トレーサビリティルール

文書とテストをゆるく接続するために、ID を持たせる。

- 業務ルール: `BR-xxx`
- ユースケース: `UC-xxx`
- 用語: `TERM-xxx`
- 非機能要件: `NFR-xxx`
- テストケース群: `T-xxx`

ただし、1対1対応を強制しすぎない。
目的は管理表を増やすことではなく、参照しやすくすること。

### 6. 変更時の更新ルール

仕様変更時は次の順で更新する。

1. 用語変更があるなら `20_ubiquitousLanguage` を直す
2. 業務ルール変更があるなら `10_businessRules` を直す
3. その振る舞いをテストで更新する
4. 必要なら実装を更新する
5. 実装都合の差分は `110_codingRules` にのみ残す

ドキュメントだけ直してテストを直さない状態を禁止する。

### 7. レビュー観点

`100_documentRules` には、文書レビュー時の観点も置くべき。

- この記述は意図の説明になっているか
- 振る舞いの詳細を文書に書きすぎていないか
- 同じ内容がテストと二重管理になっていないか
- 用語がユビキタス言語と一致しているか
- 技術都合で業務ルールを歪めていないか
- UI や DB の都合がドメイン記述に混ざっていないか

## このディレクトリにあるとよいファイル

最低限、次のファイルを置くと運用しやすい。

- `README.md`
  - このディレクトリの目的と全体ルール
- `document-placement-rules.md`
  - 何をどこに書くかの分類表
- `document-writing-rules.md`
  - 文体、粒度、禁止事項、ID 付与方針

- `review-checklist.md`
  - 文書レビュー時の確認項目

上記の初版はこのディレクトリに作成済みとする。

## 実務上の推奨

このディレクトリは、仕様そのものを増やす場所ではなく、**仕様を増やしすぎないためのガードレール**として使う。

設計の基本方針は次の一文に集約できる。

> 文書は意図と制約、テストは振る舞い、コードは実現方法を持つ。

この原則を各ルールに分解して管理すれば、DDD と DIP を崩しにくい。
