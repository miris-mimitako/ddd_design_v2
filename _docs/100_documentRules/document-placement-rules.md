# Document Placement Rules

## 目的

この文書は、情報をどこに置くかを決めるための基準である。

判断基準は次の通り。

- 意図と制約はドキュメントに置く
- 振る舞いはテストに置く
- 実現方法はコードに置く

## 配置ルール

### `00_coreBusinessPurpose`

ここには、業務の存在理由を書く。

- この機能は何のために存在するか
- 何を改善したいのか
- 失敗すると何が困るのか
- 何を成果とみなすのか

ここに書かないもの:

- 詳細なユースケース手順
- 画面項目定義
- テストケース

### `10_businessRules`

ここには、業務ルールと制約を書く。

- 禁止事項
- 許可条件
- 例外条件
- 受け入れ条件
- 条件の背景

ここに書かないもの:

- UI の実装都合
- DB スキーマ都合
- API 形式都合
- テストコードで表現できる振る舞いの列挙

### `20_ubiquitousLanguage`

ここには、用語の意味を固定する情報を書く。

- 用語名
- 定義
- 文脈ごとの意味
- 禁止する同義語
- UI 名、コード名、DB 名の対応

ここに書かないもの:

- 単なる略語一覧
- 実装クラス名の都合だけで作られた名前

### `100_documentRules`

ここには、文書運用ルールを書く。

- 何を文書に書くか
- 何を文書に書かないか
- 文書とテストの責務分離
- 文書レビューの観点
- ID 付与ルール

### `110_codingRules`

ここには、実装規約を書く。

- レイヤ責務
- 依存方向
- 命名方針
- 例外方針
- フレームワーク使用方針

### `120_testRules`

ここには、テスト方針を書く。

- 何をどの粒度でテストするか
- Domain と Application の優先順位
- テスト命名規則
- fake / stub / mock の使い分け
- 契約テストの扱い

## ドキュメントとテストの分配ルール

### ドキュメントに残すもの

- 背景
- 意図
- 制約
- 禁止事項
- 用語定義
- 非機能要件
- 監査要件
- 認可要件

### テストに寄せるもの

- 前提条件と結果の対応
- 正常系と異常系の振る舞い
- ユースケースのシナリオ
- ポート契約
- 境界値

## 迷ったときの判断

次の問いで判断する。

1. これは業務上の意味の説明か
2. これは実行して確かめるべき振る舞いか
3. これは単なる実装方法か

答えがそれぞれ次なら置き場所は決まる。

- 1 ならドキュメント
- 2 ならテスト
- 3 ならコード

## 例

### 例1: 出荷条件

- ドキュメント
  - 注文は承認済みでなければ出荷できない
  - 理由は誤出荷防止と監査のため
- テスト
  - 承認済み注文は出荷できる
  - 未承認注文は出荷できない
  - 取消済み注文は出荷できない
- コード
  - 判定ロジックの実装

### 例2: メールアドレス重複禁止

- ドキュメント
  - 同一メールアドレスでの重複登録は禁止
  - 理由は本人性と通知整合性のため
- テスト
  - 未登録メールアドレスなら登録できる
  - 既存メールアドレスなら登録できない
- コード
  - repository 参照方法

## 禁止事項

- 同じ振る舞いを文書とテストに重複して詳細記述しない
- 技術都合を業務ルールとして書かない
- 画面都合をドメイン用語に持ち込まない
- DB 名をそのままユビキタス言語にしない
