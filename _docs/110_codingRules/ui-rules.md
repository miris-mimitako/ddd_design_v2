# UI Rules

## 目的

この文書は、このプロジェクトにおける UI の構築手順と責務を定義する。

## 基本方針

- UI は backend の業務判断を再実装しない
- UI は Storybook を中心にコンポーネント単位で構築する
- ページは Story で作らず、確定済みコンポーネントを配置して組み立てる
- 最終確認は E2E で行う

## 構築手順

1. UI の元データを HTML などで受け取る
2. その構造をコンポーネント単位に分解する
3. Storybook 上で各コンポーネントを展開する
4. 不足するコンポーネントを追加実装する
5. Story 上で描画確認と必要な UI ロジック確認を行う
6. Story で確定したコンポーネントをページへ配置する
7. 画面遷移や最終挙動は E2E で確認する

## Storybook ルール

- Story はコンポーネント単位で作成する
- 画面全体の Story は作成しない
- このプロジェクトで必要な UI ロジックは Story 上で確認する
- 他プロジェクトでの再利用性は設計条件に含めない

Story 上で確認するもの:

- 描画崩れがないこと
- props による表示切り替え
- loading、disabled、error などの状態差分
- UX 上必要な入力補助
- イベント発火の結線

Story 上で確認しないもの:

- 業務ルールの最終判定
- backend の正当性
- DB 更新結果

## ページ構築ルール

- ページは Story で確定したコンポーネントを配置して構築する
- ページで新しい業務ロジックを足さない
- ページ側のロジックは composition と state 接続に限定する
- ディレクトリ責務は `frontend-structure-rules.md` に従う

## E2E ルール

- E2E は最後の確認として使う
- E2E でコンポーネント仕様の正本を持たない
- Story で確認済みの状態を、画面統合後に再確認する

## 禁止事項

- ページ完成後に初めてコンポーネント仕様を詰める
- 画面全体 Story を仕様の正本にする
- UI 側で業務判定を増やす
- 再利用性のためにこのプロジェクトの要件を曲げる
